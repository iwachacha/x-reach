# Examples

## Example 1

Rough ask:

```text
OpenAI と Anthropic の最新の agent 関連発表を比較したい
```

Normalized brief:

```markdown
調査ブリーフ
- 目的: OpenAI と Anthropic の agent 関連発表の差分と共通点を把握し、直近の動向を短時間で比較できるようにする。
- 対象: OpenAI と Anthropic の agent 関連発表、公式ドキュメント、公式ブログ、関連リポジトリ。
- 期待成果物: 比較要約。主な違い、共通点、公開日、一次ソースを短く整理する。
- 鮮度要件: 実行時点の最新情報を優先する。最終回答では具体日付を明記する。
- 含める範囲: 公式発表、公式 docs、リリースノート、関連リポジトリ、必要最小限の高信号な二次ソース。
- 除外範囲: 低信号な転載、一次ソース不明の記事、過度に古い比較記事。
- 地域・言語: 出力は日本語。調査は日英併用。
- 重視ソース: 公式ブログ、公式 docs、GitHub、メンテナー発信。
- 禁止ソース: 内容の裏取りができないまとめ記事、AI 生成ミラー。
- 証拠厳密度: standard。一次ソース優先で、差分の根拠リンクを付ける。
- 調査スケール: standard
- 前提と仮定: 比較軸の指定がなかったため、機能差分・公開日・一次ソースを中心に整理する前提で固定した。対象がグローバルなため、調査は日英併用にした。
```

## Example 2

Rough ask:

```text
この OSS の調子を調べたい
```

Normalized brief:

```markdown
調査ブリーフ
- 目的: OSS の活動状況、保守性、採用判断に関わるシグナルを把握する。
- 対象: 指定された OSS の GitHub リポジトリと公式ドキュメント。
- 期待成果物: 短い健全性サマリー。活動状況、リリース頻度、issue/PR 傾向、利用上の注意点をまとめる。
- 鮮度要件: 直近 12 か月を重視する。必要なら最新の更新日も確認する。
- 含める範囲: README、リリース、issue/PR の傾向、公式 docs、メンテナー発信。
- 除外範囲: 根拠の薄い感想、単発の外部紹介記事。
- 地域・言語: 出力は日本語。調査は英語中心で必要に応じて日本語も使う。
- 重視ソース: GitHub、公式 docs、公式ブログ、メンテナー発言。
- 禁止ソース: 裏取り不能な転載記事、内容の古い比較ページ。
- 証拠厳密度: standard。確認可能なシグナルを優先する。
- 調査スケール: standard
- 前提と仮定: リポジトリ URL または owner/name が与えられる前提で固定した。期待成果物の指定がないため、採用判断に使える健全性サマリーにした。
```

## Example 3

Rough ask:

```text
日本でも海外でもこの新製品の反応を見たい
```

Normalized brief:

```markdown
調査ブリーフ
- 目的: 新製品に対する日本語圏と英語圏の反応差と共通点を把握する。
- 対象: 指定された新製品に関する公式発表と公開コミュニティ反応。
- 期待成果物: 反応サマリー。好意的反応、懸念点、未確認情報を分けて整理する。
- 鮮度要件: 実行時点の最新反応を確認する。急速に動く話題なら具体日付を明記する。
- 含める範囲: 公式発表、Bluesky、Reddit、Hacker News、必要に応じて Twitter/X。
- 除外範囲: ボット的投稿、内容が薄い転載、根拠のない断定。
- 地域・言語: 出力は日本語。調査は日本語と英語を併用する。
- 重視ソース: 公式発表、開発元発信、日英の主要コミュニティ反応。
- 禁止ソース: 出所不明のまとめ、裏取り不能な憶測投稿。
- 証拠厳密度: standard。反応の引用は短くし、事実と感想を分ける。
- 調査スケール: standard
- 前提と仮定: 製品名は明示される前提で固定した。コミュニティ反応比較の要望があるため、日英併用を既定にした。
```
