# x_reach プロジェクト方針 v0 Draft

## 1. この文書の目的

この文書は、x_reach の実装判断・設計判断・優先順位判断に迷ったときの基準を定めるためのものである。
個別のアイデアや実装案の良し悪しを、その場の勢いや局所最適ではなく、x_reach 全体の目的に照らして判断できるようにする。

x_reach は、単に X を読めるツールではない。
**AI エージェントが、X 上でテーマに沿った大量調査を、再現可能かつ高品質に実行できるための基盤**である。

---

## 2. 背景

x_reach は、元の Agent-Reach から派生した X 特化版である。
上流 Agent-Reach は、複数媒体に対応するために、意図的に責務を絞った薄い統合基盤として設計されている。安定した JSON CLI、診断、チャネル一覧、外部統合用の read-only な収集面を提供する一方で、スケジューリングやランキング、要約、公開といった下流責務は持たない。 ([GitHub][1])

x_reach では、この「薄さ」の価値は一部維持しつつも、X 一本化したことで得られる利点を生かし、**X 調査で共通して必要になる品質向上機能は明確に作り込む**。
ただし、特定テーマ専用の硬すぎる道具にはせず、様々なテーマ・規模・ワークフローで使える汎用性は維持する。

---

## 3. x_reach の立ち位置

x_reach の立ち位置は、次の一文で表す。

**x_reach は、AI エージェント向けの X 調査実行基盤であり、単なる検索ラッパーでも、完成品の分析アプリでもない。**

x_reach が担うのは主に以下である。

* X 上の投稿収集を、機械的に安定実行できること
* 明らかなノイズを抑えられること
* テーマに合致した質の高い投稿を選びやすくすること
* 小規模調査から大規模調査まで扱えること
* 下流の分析・要約・可視化・公開フローに接続しやすいこと

x_reach が原則として担わないものは以下である。

* 調査結果の最終解釈を固定すること
* 特定テーマ専用の分析ロジックを中核に埋め込むこと
* 投稿以外の媒体の調査機能を再び広く抱え込むこと
* 「何を調べるべきか」まで完全に自動決定すること
* 完成済みのダッシュボード製品になること

---

## 4. 最上位の開発目標

x_reach の最上位目標は、以下の 4 点である。

### 4.1 質の高い収集

テーマに沿った投稿を大量に集めつつ、明らかなノイズを減らし、調査対象として有用な投稿を取りこぼしすぎないこと。

### 4.2 再現可能な調査

同じ条件で実行すれば、近い収集結果と同じ判断過程を再現しやすいこと。

### 4.3 エージェントにとって扱いやすいこと

Codex などの AI エージェントが、無駄に悩まず、安全に、安定して利用できること。

### 4.4 幅広い調査への適応性

小規模の単発調査から、大規模な継続収集・比較調査まで広く対応できること。

---

## 5. 判断に迷ったときの基本原則

### 原則1: 「X 特化で得られる価値」を優先する

上流で複数媒体対応のために抽象化されていたものでも、X に一本化した結果として明確な価値が出るなら、x_reach では具体化・最適化してよい。

### 原則2: 「テーマ依存の知能」より「X 調査共通の品質基盤」を優先する

あるテーマだけで効く特殊ロジックよりも、幅広いテーマで効くノイズ除去・重複排除・関連性判定・多様性制御・ジョブ管理を先に強くする。

### 原則3: LLM に丸投げしない

明らかなノイズ除去、重複排除、形式的な前処理、基本スコアリングは deterministic な処理または軽量なルールで先に処理する。
LLM は高コスト・高柔軟性な最終選別や曖昧判定に限定する。

### 原則4: 収集と解釈を分ける

x_reach は「集める」「整える」「選びやすくする」ことには深く関与してよい。
一方で、最終的な意味づけ・論評・公開判断は下流に委ねる。

### 原則5: API と出力は安定させる

内部実装は変えてよいが、外部から使う JSON schema、CLI 契約、主要な出力形式はなるべく壊さない。

### 原則6: 広く使えるが、薄すぎない

「汎用性を守る」ことは、「何も賢くしない」ことではない。
X 調査でほぼ常に有用な機能なら、共通基盤として持ってよい。

### 原則7: 大規模運用を先に想定する

小規模調査でしか成立しない設計は避ける。
checkpoint、resume、分割実行、失敗耐性、途中保存は早い段階で前提にする。

### 原則8: エージェントの自由度より、失敗しにくさを優先する

AI エージェントに細かい手順をその都度考えさせるより、宣言的な spec と安定 API を与えた方が、品質もコストも再現性も良くなる。

---

## 6. x_reach が強く持つべき責務

x_reach は以下の領域を、上流より一段深く持つ。

### 6.1 調査条件の固定化

エージェントの曖昧な依頼を、収集条件として固定できること。
たとえば以下を構造化できることが望ましい。

* 調査テーマ
* 期間
* 言語
* 件数目安
* 精度優先 / 網羅優先
* 除外条件
* ノイズ許容度
* 出力形式

### 6.2 収集ジョブの管理

* 大量収集を分割して実行できる
* 途中保存できる
* 再開できる
* 失敗ジョブを再試行できる
* 調査規模に応じて budget 制御できる

### 6.3 投稿の正規化

* 投稿 ID
* author
* reply / quote / repost 関係
* thread 情報
* URL / media
* 取得時刻
* クエリ由来

などを統一 schema に整える。

### 6.4 ノイズ抑制

* 明らかな宣伝
* 内容が薄い投稿
* 重複
* 同一スレッド内の過剰冗長
* 同一アカウントの偏り
* URL やハッシュタグ過多の低情報投稿

などを抑制する。

### 6.5 関連性と質の選別支援

最終判断を完全自動化する必要はないが、少なくとも以下をスコア化・付与できることが望ましい。

* テーマ関連性
* 情報量
* 具体性
* 一次体験性
* 新規性
* 多様性への寄与
* ノイズ疑い

---

## 7. x_reach が持ちすぎない責務

以下は原則として中核責務にしない。

### 7.1 テーマ固有の深い意味解釈

医療、金融、政治、製品評価など、分野固有の意味解釈まで x_reach 本体に埋め込まない。
必要ならプラグイン、ルールセット、下流処理で対応する。

### 7.2 完成された分析 UI

調査結果を閲覧する簡易出力はあってよいが、本格的な分析プロダクトやダッシュボードを本体責務にしない。

### 7.3 投稿収集以外への再拡張

X 特化で得た集中力を失わないため、他媒体対応を本体目標には戻さない。

---

## 8. 機能設計の基本方針

### 8.1 「Mission Spec First」

AI エージェントが直接コマンド断片をつなぐのではなく、まず調査条件を構造化した spec に落とし、その spec に従って x_reach が収集を実行する形を優先する。

理想形の例:

* objective
* queries
* time_range
* languages
* target_posts
* quality_profile
* exclude_rules
* diversity_rules
* outputs

これにより、調査の再現性とレビュー容易性が上がる。

### 8.2 「Raw / Canonical / Curated」の三層管理

収集結果は可能なら以下の三層で扱う。

* Raw: 取得元に近い生データ
* Canonical: 正規化済みデータ
* Curated: ノイズ除去・重複排除・スコア付与後の利用向けデータ

この分離により、ルール変更時の再評価や、後からの品質改善がしやすくなる。

### 8.3 「Deterministic First, LLM Second」

処理順は原則として以下にする。

1. 形式処理
2. ハードフィルタ
3. 重複排除
4. ルールベース scoring
5. 必要時のみ LLM で曖昧判定

### 8.4 「High Precision / Balanced / Broad Recall」の複数プロファイル

全調査に同じ設定を当てない。
少なくとも以下の 3 モードを意識する。

* High Precision: 少なめでも質優先
* Balanced: 標準
* Broad Recall: まず広く集める

## 9. ノイズ除去の方針

x_reach は、X 調査において明らかに有用な共通ノイズ対策を持つ。
ただし、過剰除外で重要投稿を失わないよう、段階的に行う。

### 9.1 ハードフィルタの対象候補

* repost / RT 系
* 極端に短い低情報投稿
* ハッシュタグ・URL 過多
* 定型懸賞・宣伝テンプレ
* 同文面の大量重複
* スパム的なアカウント由来投稿

### 9.2 ソフト判定の対象候補

* テーマ関連性が弱い
* 具体性が乏しい
* 感想のみで情報量が少ない
* 同種の投稿がすでに十分ある
* 同一 author / thread に偏っている

### 9.3 重要な原則

ノイズ除去は「削ること」自体が目的ではない。
**関連性が高く、質の良い投稿を残しやすくすること**が目的である。

---

## 10. 品質判定の方針

質の高い投稿とは、単に伸びている投稿ではない。
x_reach では、以下のような観点を重視する。

* テーマへの関連性が高い
* 具体性がある
* 一次体験または一次観測に近い
* 情報量がある
* 他の投稿と重複しすぎていない
* 調査全体の多様性に寄与する

一方で、エンゲージメントは補助指標に留める。
バズっていても調査価値が低い投稿は多く、逆に低エンゲージメントでも有用な一次情報はあるためである。

---

## 11. 多様性確保の方針

上位スコア順に並べるだけでは、同じ話題・同じアカウント・同じスレッドに偏りやすい。
そのため x_reach は、品質だけでなく多様性も結果品質の一部として扱う。

少なくとも以下の偏りは制御対象とする。

* 同一 author 偏重
* 同一 thread 偏重
* 同一 URL 偏重
* 同一観点 / 同一論点への偏り
* 特定時間帯への偏り

x_reach の目的は「最も伸びた投稿を並べること」ではなく、**調査に使える投稿集合を作ること**である。

---

## 12. AI エージェントとの役割分担

x_reach では、AI エージェントに過剰な自由裁量を持たせない。
役割分担は次を基本とする。

### エージェントが主に担うもの

* 調査目的の言語化
* 初期クエリ案の作成
* 条件の調整
* サンプル確認
* 結果の解釈
* 次の探索方針の提案

### x_reach が主に担うもの

* 収集の安定実行
* 正規化
* ノイズ抑制
* 重複排除
* スコア付け
* ジョブ管理
* 再開可能性
* 構造化出力

この分離により、エージェントは「考える」ことに集中でき、x_reach は「安定して集める」ことに集中できる。

---

## 13. 上流 Agent-Reach との関係方針

x_reach は Agent-Reach からの派生であり、上流の良さを尊重する。
ただし、上流の思想をそのまま固定的に継承するのではなく、X 特化版として必要な再定義を行う。

### 維持したいもの

* 安定した JSON 契約
* 外部から使いやすい CLI / SDK
* エージェント統合のしやすさ
* glue-layer 的な分かりやすさ
* 安全で予測可能な外部インターフェース

### x_reach で再定義するもの

* 薄さの度合い
* ノイズ除去の責務範囲
* 品質判定の責務範囲
* 大規模収集時のジョブ制御
* X 専用 schema と convenience 機能

方針として、**上流互換性のために x_reach の価値を弱めることはしない**。
ただし、外部利用者が扱いやすい安定面は壊さない。

---

## 14. 新機能の採否基準

新機能を追加するか迷ったときは、以下で判断する。

### 採用しやすい機能

* X 調査の幅広いテーマで有用
* 小規模でも大規模でも効く
* ノイズ低減または品質向上に寄与する
* 再現性や安定性を高める
* エージェント利用時の失敗率を下げる
* 下流の自由度を壊さない

### 採用しにくい機能

* 特定テーマにしか効かない
* 下流で簡単に実現できる
* 本体を重く複雑にするわりに恩恵が狭い
* エージェントの使い方を不必要に固定する
* 出力契約を不安定にする
* 調査基盤ではなく分析プロダクト寄りである

---

## 15. 完了の定義

「機能がある」ではなく、「判断材料として使える」状態を完了とみなす。
少なくとも以下を満たしていることが望ましい。

* 入出力 schema が明確
* 小規模・中規模で動作確認済み
* 途中失敗時の再開が可能
* 典型ノイズに対する挙動が確認済み
* 代表ケースで収集品質の改善が説明できる
* エージェントから安定して呼べる
* 下流で再利用しやすい形で出力できる

---

## 16. 迷ったときの最終判断ルール

設計や実装で迷ったら、最後は次の順で優先する。

1. X 調査の結果品質が上がるか
2. 大規模実行時に破綻しにくくなるか
3. エージェント利用時に安定するか
4. 再現性が上がるか
5. 幅広いテーマで使えるか
6. 下流の自由度を不必要に潰さないか
7. 実装コストに見合うか

この順で見て、上位で明確に価値があるなら採用を前向きに検討する。

---

## 17. 一文でまとめる x_reach の方針

**x_reach は、X 調査に共通して必要な品質基盤はしっかり作り込みつつ、最終解釈やテーマ固有ロジックは下流に委ねる、AI エージェント向けの再現可能な X 調査実行基盤である。**
